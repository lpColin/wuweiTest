# 五维测评系统 - 硬件规格说明书

> 版本：v1.0 | 日期：2026-06-29 | 规模：200 人同时在线

---

## 一、概述

本规格说明书针对 **五维测评系统** 在 **200 人同时在线** 场景下的硬件配置需求进行设计。系统基于 B/S 架构，包含 Web 前端、API 后端、数据库、缓存、对象存储等组件。

### 1.1 业务负载估算

| 指标 | 估算值 | 说明 |
|---|---|---|
| 同时在线用户（峰值） | 200 人 | 考虑早晚高峰 |
| 日活用户（DAU） | 800 - 1,200 人 | 在线率约 20% |
| 单次测评 API 请求量 | 60 - 300 次/人 | 每题一次提交 |
| 峰值 QPS（API） | 60 - 100 req/s | 答题集中提交场景 |
| 题库存储量 | 5,000+ 题 | 含文本、图片附件 |
| 用户数据量（年） | 约 2 GB | 用户表 + 测评记录 |
| 附件存储（年） | 约 20 GB | 海报、头像、文章封面等 |

### 1.2 系统组件

| 组件 | 技术选型 | 部署方式 |
|---|---|---|
| 前端 Web | React / Vue 3 + Vite | 静态资源部署至 Nginx |
| 后端 API | Node.js（NestJS / Express） | PM2 多进程 |
| 关系数据库 | MySQL 8.0+ / PostgreSQL 14+ | 主从架构 |
| 缓存 | Redis 6.x+ | 单机或 Sentinel |
| 对象存储 | 阿里云 OSS / MinIO | 独立服务 |
| 反向代理 | Nginx / OpenResty | SSL 终结 + 负载均衡 |

## 二、部署架构

由于 200 人同时在线规模较小，采用**单层应用 + 数据库与缓存合并**的轻量架构。MySQL 与 Redis 部署在同一台服务器上（127.0.0.1 内环回通信），减少硬件成本，200 人规模完全够用。

### 2.1 纯局域网 WiFi 部署方案

如果不依赖公网，所有用户通过局域网 WiFi 访问，使用一台路由器 + 一台服务器即可支撑 200 人规模。

```
                        ┌──────────────────────┐
                        │   服务器（有线接入）    │
                        │  MySQL + Redis + Web   │
                        │  4C8G 一台全包          │
                        └────────┬─────────────┘
                                 │ 有线（千兆）
                        ┌────────▼─────────────┐
                        │   WiFi 路由器/AP       │
                        │   支持 WiFi 6         │
                        │   建议 2-3 台 AP 组网   │
                        └──────────────────────┘
                                 │ WiFi
             ┌───────────────────┼───────────────────┐
             │                   │                    │
        ┌────▼────┐        ┌────▼────┐         ┌────▼────┐
        │ 用户设备  │  ...   │ 用户设备  │   ...   │ 用户设备  │
        │ (手机/平板)│       │ (笔记本)  │        │ (台式机)  │
        └─────────┘        └─────────┘         └─────────┘
```

#### 网络设备选型

| 设备 | 规格要求 | 估算费用 |
|---|---|---|
| **主路由器** | 千兆 WAN/LAN，支持 NAT + DHCP，带 QoS | ¥200 - ¥500 |
| **WiFi 接入点 × 2-3 台** | WiFi 6（AX3000+），每台建议 ≤ 80 设备 | ¥300 - ¥600/台 |
| **PoE 交换机**（可选） | 千兆 8 口 PoE，给 AP 供电 | ¥200 - ¥400 |
| **网线** | 六类千兆网线 | ¥50 - ¥100 |
| **合计** | - | **~¥1,000 - ¥2,000**（一次性） |

> 如果用户数 ≤ 50 人，一台高端家用 WiFi 6 路由器（如 TP-Link AX6000 / 小米 AX9000）即可搞定，成本约 ¥500。

#### 各规模 WiFi 设备推荐

| 同时在线 | AP 数量 | 推荐方案 | 设备费用 |
|---|---|---|---|
| ≤ 50 人 | 1 台 | 家用 WiFi 6 路由器 | ¥300 - ¥500 |
| 50 - 100 人 | 2 台 AP | 路由器 + 1 台 AP 扩展 | ¥800 - ¥1,200 |
| 100 - 200 人 | 3 台 AP | 千兆路由器 + 2-3 台 AP | ¥1,500 - ¥2,000 |
| 200 - 500 人 | 5+ 台 AP | 企业 AC + AP 方案 | ¥3,000+ |

#### 带宽评估

| 链路 | 速率 | 能否支撑 200 人 |
|---|---|---|
| 服务器 → 路由器 | 千兆有线（1,000 Mbps） | ✅ 远超需求 |
| 路由器 → AP | 千兆有线（1,000 Mbps） | ✅ 有余量 |
| AP → 用户（WiFi 6） | 理论 1.2 Gbps（实际 ~500 Mbps） | ✅ 足够 |
| AP → 用户（WiFi 5） | 理论 433 Mbps（实际 ~200 Mbps） | ✅ 够用 |
| **峰值并发流量** | **~100 - 150 Mbps** | **✅ 完全可支撑** |


### 2.2 单服务器极简部署（推荐局域网方案）

将 Web 服务、API、MySQL、Redis 全部部署在同一台物理/云服务器上，通过一台千兆 WiFi 路由器提供局域网无线访问，无需多节点、无需负载均衡。

```
   ┌──────────────────────────────────────┐
   │          单服务器（一台全包）            │
   │  ┌──────┐ ┌──────┐ ┌──────┐         │
   │  │Nginx │ │Node  │ │MySQL │         │
   │  │静态  │ │API   │ │+     │         │
   │  │资源  │ │服务   │ │Redis │         │
   │  └──┬───┘ └──┬───┘ └──┬───┘         │
   │     └────────┼────────┘              │
   │         127.0.0.1 内环回通信          │
   └──────────────┬───────────────────────┘
                  │ 千兆有线
         ┌────────▼────────┐
         │  WiFi 6 路由器   │
         │  + AP（建议2-3台）│
         └────────┬────────┘
                  │ WiFi
        ┌─────────┼─────────┐
        │         │         │
    ┌───▼───┐ ┌───▼───┐ ┌───▼───┐
    │ 用户 1 │ │ 用户 2 │ │ 用户 N │
    │手机/   │ │笔记本  │ │平板   │
    └───────┘ └───────┘ └───────┘
```

**适用场景：** 校园机房、企业培训室、军营内部网络等纯局域网环境，无需公网。
**优势：** 硬件成本最低、运维最简单、一台搞定。

#### 结论

**支持，而且绰绰有余。** 对于 200 人规模：

1. **服务器** 1 台 4C8G 全包（Web + MySQL + Redis），有线接入路由器即可
2. **路由器** 千兆家用/企业级均可，WiFi 6 推荐
3. **AP 数量** 建议 2-3 台 WiFi 6 接入点，每台分担约 70-80 设备
4. **网线** 服务器 → 路由器 → AP 全部用千兆有线连接
5. **无需公网 IP**，所有设备在同一个局域网段内访问

> **最低成本方案**（≤ 50 人）：一台 ¥500 的 WiFi 6 路由器 + 一台 ¥400/月的服务器，总投入不到 ¥1,000 即可跑起来。

## 三、服务器硬件规格

### 3.0 单服务器极简方案（一台全包）

| 项目 | 规格 |
|---|---|
| **用途** | Nginx 静态资源 + Node.js API + MySQL 8.0 + Redis 6.x |
| **数量** | 1 台 |
| **CPU** | 4 核（x86_64，2.5 GHz+） |
| **内存** | 8 GB（Nginx/Node 1 GB + MySQL 5 GB + Redis 2 GB + 余量 1 GB） |
| **系统盘** | 40 GB SSD |
| **数据盘** | 100 GB SSD |
| **操作系统** | CentOS 7.9+ / Ubuntu 20.04+ |
| **网络** | 千兆网卡，有线接路由器 |
| **部署方式** | Nginx 反代 Node.js（127.0.0.1:3000），MySQL+Redis 同机 127.0.0.1 通信 |
| **单台月估费** | **~¥300 - ¥500**（云服务器或物理机）|

**性能说明：** 4C8G 一台机器跑全部服务，200 人同时在线下 CPU 负载约 40-60%，内存占用约 6 GB，完全在安全范围内。如 CPU 持续 > 80%，说明硬件需要升级，但 200 人达不到这个瓶颈。

### 3.1 应用/Web 服务器（3 节点）

| 项目 | 规格 |
|---|---|
| **用途** | 运行 Web 前端静态资源 + API 后端（Node.js） |
| **数量** | 3 台 |
| **CPU** | 2 核（x86_64，2.0 GHz+） |
| **内存** | 4 GB |
| **系统盘** | 40 GB SSD |
| **数据盘** | 无需额外数据盘 |
| **网络** | 千兆网卡 |
| **操作系统** | CentOS 7.9+ / Ubuntu 20.04+ |
| **部署内容** | Nginx + Node.js（PM2 管理） |
| **单台月估费** | ~¥120 - ¥150（云服务器） |

**说明：** 3 节点通过 Nginx upstream 做负载均衡，单节点故障不影响服务。200 人同时在线下单节点承载约 70 峰值请求，CPU 和内存余量充足。

### 3.2 反向代理 / 负载均衡

| 项目 | 规格 |
|---|---|

| **部署方式** | 复用 3.1 中一台节点，无需独立硬件 |
| **软件** | Nginx / OpenResty |

### 3.3 数据库与缓存合并服务器（MySQL + Redis）

| 项目 | 规格 |
|---|---|
| **用途** | 用户数据、题库、测评记录、积分 + Redis 缓存服务 |
| **数量** | 1 台 |
| **CPU** | 4 核（x86_64，2.5 GHz+） |
| **内存** | 8 GB |
| **系统盘** | 40 GB SSD |
| **数据盘** | 100 GB SSD（高 IOPS，建议 NVMe） |
| **网络** | 千兆网卡 |
| **操作系统** | CentOS 7.9+ / Ubuntu 20.04+ |
| **数据库版本** | MySQL 8.0+ / PostgreSQL 14+ |
| **Redis 版本** | Redis 6.x+（同机部署，127.0.0.1:6379） |
| **MySQL 参数** | innodb_buffer_pool_size = 4 GB，max_connections = 500 |

> **说明：** MySQL 与 Redis 合并部署在同一台服务器，通过 127.0.0.1 环回地址通信。内存分配为 MySQL 5 GB + Redis 2 GB + 系统余量 1 GB，200 人同时在线规模完全够用。当用户量超过 500 时，再将 Redis 拆分至独立服务器。

**存储估算（年增量）：**

| 数据类型 | 年增量 |
|---|---|

| 测评记录（5 万条/年） | ~800 MB |

| 运营数据（文章/公告等） | ~50 MB |
| 日志及临时数据 | ~300 MB |
| **合计（含索引）** | **~2 GB/年** |

### 3.4 Redis 缓存（合并部署在数据库服务器上）

| 项目 | 规格 |
|---|---|
| **部署方式** | 与 MySQL 同机部署，共用 3.3 节合并服务器 |
| **通信地址** | 127.0.0.1:6379（内环回，无网络延迟） |
| **内存分配** | maxmemory 2 GB（服务器 8 GB 总内存中分配） |
| **持久化** | RDB 快照 + AOF 追加文件 |
| **淘汰策略** | allkeys-lru |
| **适用场景** | Session 缓存、随机抽题结果缓存、验证码、计数器 |

**缓存内容估算：** 用户 Session（~20 MB）+ 抽题缓存（~100 MB）+ 其他（~30 MB），峰值小于 500 MB。

## 六、性能指标与验收标准

| 指标 | 目标值 | 测试方式 |
|---|---|---|
| 页面首屏加载时间 | ≤ 2 秒 | Lighthouse / Chrome DevTools |
| API 响应时间（P95） | ≤ 500 ms | wrk / autocannon 压测 |
| 随机抽题响应 | ≤ 200 ms | 依赖 Redis 缓存命中率 ≥ 95% |
| 测评提交处理耗时 | ≤ 1 秒 | 含评分计算和存储写操作 |
| 报告生成耗时 | ≤ 3 秒 | 含雷达图渲染 |
| 并发支撑能力 | ≥ 200 同时在线的稳定运行 | JMeter / Locust 压测 |
| 系统可用性 | ≥ 99.9%（年度） | 多节点容错 + 健康检查 |
| 数据备份 RPO | ≤ 24 小时 | 每日全量备份 + Binlog |
| 数据恢复 RTO | ≤ 2 小时 | 从库切换或备份恢复 |

## 七、扩展性说明

| 在线用户 | 操作 | 月估费用 |
|---|---|---|
| 200（单服务器） | **一台全包**（4C8G），单机 Nginx+Node+MySQL+Redis + WiFi 路由器 | **一次投入 ¥4,000 - ¥6,000**（无月费）|
